**Phase goal**: Establish the full Obsidian plugin scaffold, the settings tab skeleton, and a stubbed `BlockDetector` module. At the end of this phase, the plugin loads into Obsidian without errors, displays a settings tab, and has source files ready for Phase 2 implementation.

### 1.5 esbuild.config.mjs

Standard Obsidian esbuild config adapted from community convention. Key elements:

```javascript
import esbuild from "esbuild";
import process from "process";
import builtins from "builtin-modules";

const banner = `/*
THIS IS A GENERATED/BUNDLED FILE BY ESBUILD
if you want to view the source, please visit the github repository
*/
`;

const prod = process.argv[2] === "production";

const context = await esbuild.context({
  banner: { js: banner },
  entryPoints: ["src/main.ts"],
  bundle: true,
  external: [
    "obsidian",
    "electron",
    "@codemirror/autocomplete",
    "@codemirror/collab",
    "@codemirror/commands",
    "@codemirror/language",
    "@codemirror/lint",
    "@codemirror/search",
    "@codemirror/state",
    "@codemirror/view",
    "@lezer/common",
    "@lezer/highlight",
    "@lezer/lr",
    ...builtins,
  ],
  format: "cjs",
  target: "es2018",
  logLevel: "info",
  sourcemap: prod ? false : "inline",
  treeShaking: true,
  outfile: "main.js",
  minify: prod,
});

if (prod) {
  await context.rebuild();
  process.exit(0);
} else {
  await context.watch();
}
```

**Key points**:
- All CodeMirror 6 modules and Obsidian itself are marked `external` -- they are provided at runtime by Obsidian.
- Node builtins (`child_process`, `path`, `fs`, `crypto`, `os`) are external via the `builtin-modules` package.
- Output is a single `main.js` CJS bundle, placed at the project root (required by Obsidian).
- Watch mode for development (`npm run dev`).

### 1.8 README.md

```markdown
# Obsidian LaTeX Algorithms Render

Render ` ```latex-algorithm` code blocks as images when the cursor is outside the block. Shows raw code when the cursor is inside the block.

Requires: TeX Live / MacTeX, poppler-utils (pdftoppm) or ImageMagick.

## Usage

Wrap your LaTeX algorithm in a code fence with the `latex-algorithm` language tag:

    ```latex-algorithm
    \begin{algorithm}
    \caption{My Algorithm}
    \begin{algorithmic}
    \STATE ...
    \end{algorithmic}
    \end{algorithm}
    ```

## Development

- `npm run dev` -- watch mode
- `npm run build` -- production build
- `npm run typecheck` -- type-check without emitting
```

### 3.3 SettingTab Class

`LatexAlgoSettingTab extends PluginSettingTab` with a `display()` method that renders a settings form containing:

1. **Rendering** section:
   - DPI (number input, 50--600 range)
   - Padding (number input, 0--100 px)
   - Compile timeout (number input, 5--120 seconds)
   - Fallback engine (dropdown: pdftoppm / imagemagick)

2. **LaTeX** section:
   - Extra preamble (multi-line textarea for additional `\usepackage{}` commands)

3. **Cache** section:
   - Cache TTL (number input, 1--8760 hours)
   - Cache directory (text input, with placeholder showing default path)
   - "Flush cache" button (disabled in Phase 1 -- no-op button that logs to console)

4. **Behaviour** section:
   - Show raw by default (toggle)

All settings immediately call `plugin.saveSettings()` on change.

### 3.4 Settings Lifecycle in main.ts

```typescript
export default class LatexAlgoRenderPlugin extends Plugin {
  settings: LatexAlgoSettings;

  async onload() {
    await this.loadSettings();
    this.addSettingTab(new LatexAlgoSettingTab(this.app, this));
    // (future: register editor extensions, post processors, etc.)
  }

  async loadSettings() {
    this.settings = Object.assign({}, DEFAULT_SETTINGS, await this.loadData());
  }

  async saveSettings() {
    await this.saveData(this.settings);
  }
}
```

**Design rationale**: `Object.assign` with `DEFAULT_SETTINGS` as target ensures that new settings added in later versions automatically get their default value for existing users.

```typescript
export interface CodeBlockRange {
  /** 0-indexed line of the opening fence (```latex-algorithm) */
  fromLine: number;
  /** 0-indexed line of the closing fence (```) */
  toLine: number;
  /** Raw text content between the fences (excludes the fence lines) */
  source: string;
  /** The language tag (e.g. "latex-algorithm") */
  language: string;
}

export interface BlockDetector {
  /**
   * Scan the given document text and return all latex-algorithm code blocks.
   * Used for initial discovery and full re-scans.
   */
  findAll(doc: string): CodeBlockRange[];

  /**
   * Return the block that contains the given line number, or null.
   * Used for cursor-position lookups.
   */
  findBlockAtLine(doc: string, line: number): CodeBlockRange | null;
}
```

## 6. Integration Wiring in main.ts

Complete `main.ts` after Phase 1:

```typescript
import { Plugin } from "obsidian";
import { LatexAlgoSettings, DEFAULT_SETTINGS, LatexAlgoSettingTab } from "./settings";
import { BlockDetector, DefaultBlockDetector } from "./editor/block-detector";
import { ProcessRunner, DefaultProcessRunner } from "./utils/process";
import { TempDirManager, DefaultTempDirManager } from "./utils/tempdir";

export default class LatexAlgoRenderPlugin extends Plugin {
  settings: LatexAlgoSettings;
  blockDetector: BlockDetector;
  processRunner: ProcessRunner;
  tempDirManager: TempDirManager;

  async onload() {
    console.log("[LaTeX Algorithms Render] Loading plugin...");

    await this.loadSettings();

    this.blockDetector = new DefaultBlockDetector();
    this.processRunner = new DefaultProcessRunner();
    this.tempDirManager = new DefaultTempDirManager();

    this.addSettingTab(new LatexAlgoSettingTab(this.app, this));

    console.log("[LaTeX Algorithms Render] Plugin loaded successfully.");
  }

  onunload() {
    console.log("[LaTeX Algorithms Render] Plugin unloaded.");
  }

  async loadSettings() {
    this.settings = Object.assign({}, DEFAULT_SETTINGS, await this.loadData());
  }

  async saveSettings() {
    await this.saveData(this.settings);
  }
}
```

```bash
#!/usr/bin/env bash

VAULT_DIR="$1"

  echo "Usage: $0 <vault-directory>"
  exit 1
fi
PLUGIN_DIR="$VAULT_DIR/.obsidian/plugins/latex-algorithms-render"
mkdir -p "$PLUGIN_DIR"
cp main.js manifest.json styles.css "$PLUGIN_DIR/"
echo "Installed to $PLUGIN_DIR"

### 7.4 Vault-side verification

1. Enable the plugin in Obsidian Settings > Community Plugins.
2. Confirm the plugin appears in the list as "LaTeX Algorithms Render".
3. Open plugin settings -- all setting controls render without errors.
4. Toggle each setting, reload vault, confirm values persist.
5. Check Obsidian DevTools (Cmd+Opt+I) console for the `[LaTeX Algorithms Render]` log messages.
6. (Negative test) Create a ` ```latex-algorithm` block and confirm it shows as raw code (no crash, no image -- rendering comes in Phase 3).

### 7.5 Error Cases

| Test case | Expected behaviour |
|-----------|-------------------|
| Missing `main.js` | Obsidian shows "Failed to load plugin" in settings |
| Invalid `manifest.json` syntax | Obsidian ignores the plugin entirely |
| Settings corrupted in `data.json` | `Object.assign` with defaults recovers gracefully |
| Plugin loaded twice | Obsidian prevents duplicate `onload()` automatically |

## 9. Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Obsidian API version mismatch (`minAppVersion` too low/high) | Set `minAppVersion` to `1.0.0` (well-established baseline). Update if specific API features require a newer version. |
| esbuild version incompatibility with Node.js version | Pin `esbuild@0.17.3` (known to work). The project uses system Node.js which is >= 18. |
| `builtin-modules` package missing a Node built-in | Verify that `child_process`, `path`, `fs`, `crypto`, `os` are all in the list. If not, add them manually to the `external` array. |
| Plugin conflicts with other renderers | The `latex-algorithm` language tag is specific enough that collision is unlikely. No other known plugin uses this tag. |
